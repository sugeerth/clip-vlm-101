One in a million

Everything else on this site runs on a gallery of 14 images. This page is what happened when we grew the same database design to 690,000 real dishes (of the dataset's 1M+, the slice this laptop has ingested so far) — Qdrant's public Wolt food dataset, image embeddings precomputed with the same clip-ViT-B-32 checkpoint the browser demo runs, plus 690,000 text embeddings we computed ourselves, one dish name at a time, on one laptop. Same 512-d space. Same dot products. Honest numbers — the shape lessons are identical at any scale.

Nothing on this page is a simulation — every number below was measured by scale.py bench on an Apple-silicon laptop, and the machinery is CI-tested synthetically on every push.

Search it — right here, right now

A 100,000-dish slice of the corpus, optimised-product-quantized to 64 bytes per dish (pq.py → js/pq.js): the whole index is a ~7 MB download, your query is embedded by the same CLIP text tower the demo uses, and the scan is pure table lookups in this tab — no server anywhere. Want the exact top-10? Tap ＋ precision to stream a 51 MB int8 tier and re-rank on top. First search downloads the model (~65 MB) + the pack, then it's milliseconds.

What changes at a million is the shape, not the math

db.py stores every vector as a float32 BLOB inside SQLite — perfect at 14 rows, wrong at a million, because a brute-force scan would decode a million BLOBs per query. At scale, every real vector store splits the two jobs. So does scale.py: 

data/scale.sqlite       the RECORDS — name, caption, cafe, url.
                        row id i here IS row i of every matrix below.

data/img_emb_f16.npy    the SCAN — one packed (690k × 512) float16 matrix
data/txt_emb_f16.npy    per tower, memory-mapped: a query is a chunked
                        matrix @ vector, not 690k BLOB decodes.

The searches build on concepts the repo already taught: brute is search.py's scan, chunked; ivf is ann.py's inverted-file index trained on a 100k sample; int8 is quantize.py's lesson, now with a per-dimension scale; and OPQ (opq.py) shrinks a vector to 64 bytes — a coarse filter that a short exact re-rank turns back into the truth. Fused search never materialises a 1024-d matrix at all — fusion.py's identity, score = (image·q + text·q) / 2, applied lazily across two towers.

Sizes first — 690k rows fits in a coat pocket, and the compressed indexes in a much smaller one:

Compression vs. accuracy — recall@{1,10,100} against the exact top-k, self-queries (the hardest case), measured by scale.py bench:

methodbytes/vecrecall@1recall@10recall@100
brute force, f32 (the truth)20481.001.001.00

The bottom row is the headline: 64 bytes a vector — 32× smaller than float32 — and recall@10 of 0.92. Coarse OPQ keeps only 0.59 of the exact top-10 (compression showing), but its top few hundred almost always contain the true neighbours, so a short exact re-rank recovers them. Two lessons the 14-image gallery could never teach: a per-dimension int8 scale beats one global scale on CLIP's outlier dimensions (0.86 vs the old 0.68), and OPQ's learned rotation (opq.py) earns ~10 recall points over plain product quantization for free — same 64 bytes, just a better split.

Speed — the storage hierarchy, exact answers throughout (median over queries, warm cache):

where the bytes livems / queryrecall@10
f16 memmap (paged off disk)1101.00 — the truth
f32 in RAM (serve)18.61.00 — the truth
f32 on the GPU / MPS (serve)12.31.00 — the truth

The memmap scan was never really about math — it was paging a memory-mapped file off disk and casting f16→f32, query after query. python3 scale.py serve promotes both towers to f32 RAM once (~2.8 GB) and the same exact scan is one matmul; the GPU is one more rung. Same answers, an order of magnitude apart — change where the bytes live, not the algorithm.

Recall climbs, then flattens — a few true neighbours live in cells no nearby probe visits, and buying them back gets expensive. 16 probes keep 0.90 of the truth while touching under 2% of the rows.

Reproduce it

The vectors stay out of git (4 GB of parquet in, ~2.5 GB of database out) — the repo ships the machinery, and CI proves it on synthetic data every push (python3 scale.py selftest). To rebuild this page's numbers: 

# 20 parts × 50k dishes from Qdrant's public dataset (~4.4 GB)
mkdir -p data
for i in $(seq -w 0 19); do
  curl -L -o data/part-$i.parquet https://huggingface.co/datasets/\
Qdrant/wolt-food-clip-ViT-B-32-embeddings/resolve/main/data/\
part-$i-wolt-clip-ViT-B-32.parquet
done

python3 scale.py ingest        # parse, normalise, embed every dish name
                               # through the text tower (~500 rows/s)
python3 scale.py bench         # builds int8 + OPQ + ivf, prints the honest table
python3 scale.py search "quattro formaggi pizza" --method opq   # 64B → re-rank
python3 scale.py serve         # live at localhost:8071 — towers in RAM,
                               # fused + query-ensembled, cost on every answer

Dataset: Qdrant / wolt-food-clip-ViT-B-32-embeddings — 1.75M dishes from the Wolt app, embeddings by the clip-ViT-B-32 checkpoint. We take the first 20 parts and compute the text tower ourselves (embedder.py, ~1,900 names/s on MPS). Images stay on Wolt's CDN; we store URLs, not pixels.
