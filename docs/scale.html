One in a million

Everything else on this site runs on a gallery of 14 images. This page is what happened when we grew the same database design to 1,000,000 real dishes — Qdrant's public Wolt food dataset, image embeddings precomputed with the same clip-ViT-B-32 checkpoint the browser demo runs, plus a million text embeddings we computed ourselves, one dish name at a time, on one laptop. Same 512-d space. Same dot products. Honest numbers.

Nothing on this page is a simulation — every number below was measured by scale.py bench on an Apple-silicon laptop, and the machinery is CI-tested synthetically on every push.

Search it — right here, right now

A 100,000-dish slice of the million (every 10th row), product-quantized to 64 bytes per dish (pq.py → js/pq.js): the whole index is a ~7 MB download, your query is embedded by the same CLIP text tower the demo uses, and the scan is pure table lookups in this tab — no server anywhere. First search downloads the model (~65 MB) + the pack, then it's milliseconds.

What changes at a million is the shape, not the math

db.py stores every vector as a float32 BLOB inside SQLite — perfect at 14 rows, wrong at a million, because a brute-force scan would decode a million BLOBs per query. At scale, every real vector store splits the two jobs. So does scale.py: 

data/scale.sqlite       the RECORDS — name, caption, cafe, url.
                        row id i here IS row i of every matrix below.

data/img_emb_f16.npy    the SCAN — one packed (1M × 512) float16 matrix
data/txt_emb_f16.npy    per tower, memory-mapped: a query is a chunked
                        matrix @ vector, not a million BLOB decodes.

The searches are the ones the repo already taught, unchanged: brute is search.py's scan, chunked; ivf is ann.py's inverted-file index trained on a 100k sample; int8 is quantize.py's lesson applied to the whole matrix. Fused search never materialises a 1024-d matrix at all — fusion.py's identity, score = (image·q + text·q) / 2, applied lazily across two towers.

The last two rows are the punchline: the 868 ms scan was never really about math — it was paging a memory-mapped gigabyte off disk and casting it, query after query. python3 scale.py serve promotes both towers to f32 RAM once (~4 GB, under a second from a warm file cache) and the same exact scan becomes one matmul: 32× faster, identical answers. That is the storage-hierarchy lesson at 1M — change where the bytes live, not the algorithm.

And the whole point of ann.py, now a thousandfold louder — recall is a dial, priced in rows scanned:

Recall plateaus at 0.95 — a few true neighbours live in cells the probes never visit, and buying them back gets expensive. Recall is a dial, priced in rows scanned. The int8 line is the other half of the lesson: one shared scale is too coarse for CLIP's outlier dimensions (0.68 on its own), but quantisation noise rarely knocks the truth out of the top-100 — fetch cheap, score exactly, and 0.68 becomes 0.91 at the same speed.

Reproduce it

The vectors stay out of git (4 GB of parquet in, ~2.5 GB of database out) — the repo ships the machinery, and CI proves it on synthetic data every push (python3 scale.py selftest). To rebuild this page's numbers: 

# 20 parts × 50k dishes from Qdrant's public dataset (~4.4 GB)
mkdir -p data
for i in $(seq -w 0 19); do
  curl -L -o data/part-$i.parquet https://huggingface.co/datasets/\
Qdrant/wolt-food-clip-ViT-B-32-embeddings/resolve/main/data/\
part-$i-wolt-clip-ViT-B-32.parquet
done

python3 scale.py ingest        # 1M rows: parse, normalise, embed a million
                               # dish names through the text tower (~35 min)
python3 scale.py search "quattro formaggi pizza" --mode fused
python3 scale.py bench --ram --queries "sushi,burger,ramen,pad thai"
python3 scale.py serve         # the million live at localhost:8071 —
                               # fused + query-ensembled, cost on every answer

Dataset: Qdrant / wolt-food-clip-ViT-B-32-embeddings — 1.75M dishes from the Wolt app, embeddings by the clip-ViT-B-32 checkpoint. We take the first 20 parts and compute the text tower ourselves (embedder.py, ~1,900 names/s on MPS). Images stay on Wolt's CDN; we store URLs, not pixels.
