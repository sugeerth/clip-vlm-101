System design · a blog post

Fourteen images to a billion.

The rest of this site searches 14 images. One page grew the same database design to a real million and timed it on a laptop. This post finishes the arc: what it takes to serve a billion — and why the answer is not a rewrite, but the same two towers, an index that costs O(√N) instead of O(N), sixty-four bytes a vector, and a thousand copies of the million we already measured. Every number below is either measured by scale.py or arithmetic by scaling.py, which CI re-runs on every push. Nothing is a vibe.

Read time ~8 minutes. Companion to the terse ARCHITECTURE.md map.

01 · The load-bearing choice

Two towers, one space — the decision that makes scale possible

Everything downstream is affordable because of one architectural choice made at the very top. CLIP has two encoders — a vision tower and a text tower — trained so that an image and its caption land at the same spot in one 512-dimensional space. The towers never see each other; they meet only at a dot product. That independence is not an accident, it is the whole game.

Item tower — OFFLINEEvery stored image is embedded once, before any query exists. A billion forward passes, run in batch on a fleet, then never again. The output is a matrix of vectors — item_tower.py.
Query tower — ONLINEAt search time, exactly one forward pass: the query → a 512-d vector. Then it is pure arithmetic against the precomputed matrix — user_tower.py.

Ranking every item for a query is then item_matrix @ query_vec — one matrix multiply, no image model on the serving path. That is the sentence the next four sections cash out.

Contrast this with a cross-encoder, which feeds (query, item) together through one network for a richer score. It is more accurate — and it is a non-starter at scale, because it must run the model once per candidate, per query. A billion items would mean a billion forward passes at query time. The two-tower split moves all of that work to the offline side, so the online side only ever does dot products. We pay for the accuracy we lost with a cheap re-ranking stage later (§the funnel) — the industry's answer, and this repo's: dcn.py for the interaction the towers can't express, on a few hundred candidates, not a billion.

02 · The shape of every real system

Billions → hundreds → ordered → explained

A production visual search is a funnel: each stage is cheaper per item and richer per item than the last, and each hands a shorter list to the next. This repo has one file per stage — the same design at 14 rows and at 1,000,000,000, only the layout changes.

stagewhat it doesN in → outthe file
0 · Encodetwo towers → one query vector1 query embedder.py · fusion.py
1 · Retrievecheap ANN over compressed vectors — recall, not order 10⁹ → ~10²ann.py · pq.py
2 · Rankrich query×item interaction the dot product can't express ~10² → ordereddcn.py · learn2rank.py
3 · Explain + gatesay why, guarantee coverage, or abstain top-k → trustedexplain.py · trust.py

The expensive, accurate models only ever see the hundreds that survive stage 1. That is why stage 1 — retrieval — is where scale is won or lost, and it is the rest of this post.

03 · The back of the envelope

A billion vectors, three honest constraints

scaling.py does no simulation — it is pure arithmetic over three facts the rest of the repo already earned. Here is its output for N = 1,000,000,000 vectors at 512 dimensions, annotated.

① Memory — sixty-four bytes changes everything

encodingbytes / vectora billionwhere
float3220481.9 TBdb.py's raw BLOB
float161024953.7 GBscale.py's packed matrix
int8512476.8 GBquantize.py
PQ-646459.6 GBpq.py — what ships

Product quantization (pq.py) splits each 512-d vector into 64 eight-dim subvectors and replaces each with the one byte naming its nearest codebook centroid. A billion vectors go from 1.9 terabytes to 60 gigabytes — a handful of commodity boxes, and small enough that the million-row page ships a 100k slice of it into your browser.

② Work — the index is O(√N), not O(N)

Exact search is one dot product per stored vector — a billion multiplies per query, hopeless. IVF (ann.py) clusters the corpus into nlist ≈ √N cells and scans only the nprobe nearest ones. Because embeddings cluster — animals near animals, the demo's map shows it — a query's true neighbours nearly all live in a few cells.

N = 1,000,000 (measured)N = 1,000,000,000 (arithmetic)
cells (√N)1,00031,623
avg cell holds1,00031,623
nprobe = 32 scans32,0001,011,922
fraction of corpus3.2%0.1012%

The punchline is the last row: 1000× more data, but the query scans a smaller fraction of it. A billion-vector query touches ~1.04 million vectors — the √N centroids plus one probe's worth of candidates — not a billion. Retrieval work grows like the square root of the corpus. That single fact is why vector search exists.

③ Sharding — a billion is 1,000 copies of a measured million

Here is the honest bridge from a laptop to a billion. We measured a million rows on one machine (§next). A billion is simply 1,000 shards of that million, each a 61 MB PQ index that fits in one box's RAM. The query scatters to all shards in parallel, each returns its local top-k, and a tiny gather step merges them.

      query vector
           │  scatter (fan out, in parallel)
     ┌─────┼─────┬─────┬───   …   ─┬─────┐
   shard₀ shard₁ shard₂ shard₃    shard₉₉₉      each = the measured million
     │     │     │     │           │            (61 MB PQ, IVF, in RAM)
     └─────┴──┬──┴─────┴───  …  ───┴─────┘
              │  gather (merge 1,000 local top-k)
        the global top-k

Wall-clock latency is one shard's time (they run in parallel), plus a merge and a network hop — not 1,000× the work. The per-shard number is real, measured, on a laptop. The fan-out is the oldest trick in distributed systems. That is the whole trick to a billion: don't build a bigger index, build a thousand of the one that already works.

04 · What a query actually costs

The latency budget — interactive, at a billion

System design is spending a millisecond budget. Here is where a single query's time goes, composed by scaling.py from measured per-stage anchors. Shards run in parallel, so the scan is counted once — the slowest shard, not the sum.

Each bar is a stage; the total is a query's p50 at a billion vectors.

These anchors are not invented — they are the million-row page's own laptop measurements. The per-shard IVF scan is budgeted at 6 ms; scale.py bench measured the same million-row IVF index at 4.0 ms in RAM (and 11 ms from a memory-mapped file — the difference was pure I/O and f16→f32 casting, ~90% of a naive 868 ms brute scan). The query-tower forward pass, ~12 ms, is the only model on the path and the single biggest line item — which is exactly why the two-tower split, keeping the item tower offline, is what buys the budget.

05 · ML systems and agents, hand in hand

The agent layer rides on top — for free

Every agent in this repo is placed so that it does not inflate the p50 above. That is the design discipline: intelligence you can afford is intelligence that runs where it's earned, not on every query.

agentwhere it runscost on the query path
crawler.py · spider.pyOFFLINE — grows the corpus zero — never on the query path
item_tower.py + agent.pyOFFLINE — embeds & verifies zero — the towers ran before you asked
hermes.pyONLINE — refines the query usually zero extra; see below
trust.py · judge.py · drift.py OFFLINE — evaluation & monitoringzero — sampled, not per-query

The subtle one is Hermes, the agentic searcher, because it runs online. It treats your query as a draft: it proposes a few phrasings (a cheap batch through the query tower), ranks with each, and then can refine — blend the query toward its own top hits and search again. Naïvely, that multiplies the latency by the number of passes. So Hermes has an evaluator — anchored to your original query — that judges every pass: a better one is accepted, a worse one is rejected and the loop stops.

Measured, before it shipped: unguarded feedback drifts — recall fell from 0.348 to 0.333 on a 5,000-vector corpus as blind passes wandered away from the question. The evaluator guard never loses ground and settles in ~2 passes. On the live gallery you can watch it decide: the pizza query's refinement pass drifts and is rejected; the dog query converges. See it live →

Because most queries pass the gate on the first try, the p50 latency in §4 is unchanged by adding Hermes. Only the genuinely ambiguous queries spend a second search — the system pays for extra intelligence exactly where it helps, and nowhere else. Self-correction, in a latency budget, means knowing when to stop.

06 · Nothing here is a vibe

Reproduce every number

Two commands. The first is pure arithmetic (this whole post); the second is the laptop measurement it's anchored to. CI runs both on every push.

$ python3 scaling.py          # the billion-row budget, printed — no model, no data
$ python3 scale.py bench      # the million-row measurement it extrapolates from

The billion-scale arithmetic — memory linearity, the √N index law, the 1,000-shard plan, the parallel latency budget — is asserted in test_smoke.py (test_scaling) and re-run in CI, right next to scale.py selftest, so the story and the code can never drift apart.

That's the system, end to end: two towers so the work is offline, an O(√N) index so retrieval is sublinear, 64 bytes so a billion fits in RAM, a thousand shards so a laptop's measurement is the fleet's, a millisecond budget so it stays interactive, and an agent layer arranged so intelligence never costs the p50. Fourteen images taught it; a billion only changes the layout.
