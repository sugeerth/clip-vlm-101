See what a vision-language model sees.

CLIP maps images and sentences into the same 512-dimensional space, so “how similar are this photo and this phrase?” is a single dot product. This page walks the whole pipeline — embeddings, zero-shot tags, dynamic multi-label classification, a self-critiquing agent — and every demo runs live in your browser. Nothing you type or upload leaves your machine.

Try the search Open the Lab Read the code

Model: openai/clip-vit-base-patch32, int8-quantized via transformers.js. Loads on first use (~65 MB text · ~90 MB vision), cached by your browser after that.

01 · The idea

One space, two encoders — trained so the diagonal lights up

CLIP was trained contrastively on 400 million (image, caption) pairs: pull each image toward its own caption, push it away from every other. Below is that idea, live — each of the 14 gallery images scored against each of the 14 stored captions using the exact vectors in the database (no model needed: the embeddings were computed at ingest time). Darker = more similar. The bright diagonal is the training objective, visible.

Hover or tap any cell — rows are images, columns are their captions. Every cell is one dot product: image_emb · text_emb.
less similarmore similar

Raw CLIP cosines are small numbers (matches land around 0.2–0.35) — that is why the model ships a learned temperature of ~100 to sharpen them before any softmax. Section 06 uses it.

03 · The database

Fourteen images, embedded and auto-tagged at ingest

Every row in gallery.sqlite stores a path, a caption, meta tags, and three float32 vectors (512 + 512 + 1024). Click any image to open it in the Lab and inspect its actual numbers.

04 · The embedding space

All 512 dimensions, squashed onto your screen

PCA projects each image embedding to two dimensions — images that look alike land close together. Hover to see each image’s nearest neighbours in the full 512-d space (thin lines); click to open it in the Lab. Upload something in the Lab and it appears here too.

The two axes are the top principal components — directions of most variation, not nameable concepts. Distances here approximate (poorly but usefully) distances in 512-d.

05 · Search

Type a sentence, rank the database

Your query goes through the text encoder; every stored vector is compared with one dot product. Switch the mode to feel the difference between matching what images look like and what they mean.

average of visual + semantic similarity — one vector, both signals

The CLIP text encoder loads on your first search (~65 MB, cached after that).

06 · The Lab

Pick an image. Watch it become data.

Choose a gallery image or upload your own, then explore its full record: multi-class tags vs a dynamic multi-label set (drag the threshold!), the embedding agent auditing its own work, and the raw vectors that end up in the database.

Multi-class: top-5 tags

argsort the scores, keep 5 — every image gets exactly five, no matter what’s in it (tagger.py)

Multi-label: every tag decides for itself

per-tag sigmoid against the neutral prompt “a photo” — the label set is sized by the image (labels.py)

threshold 0.50

The embedding agent: propose ⇄ critique

Each round drafts labels + a caption through one prompt template, then a critic checks the work: does the caption’s embedding point back at the image (aligned ≥ 0.20)? Are the labels sure of themselves (confident ≥ 0.60)? Only a record that passes gets published (agent.py · item_tower.py).

07 · Optimize it

Same model, same weights, better answers

Everything above uses one checkpoint. These four levers — all implemented in the Python pipeline — make it more accurate or much faster without training anything.

Prompt ensembling

Embed each tag through many phrasings — “a photo of…”, “a drawing of…”, “a close-up of…” — and average the unit vectors. Phrasing noise cancels; meaning stays. ensemble.py

+3.5% zero-shot accuracy (CLIP paper, 80 templates)
Batching

One forward pass per tower per chunk of images, instead of two round-trips per image. The encoders are happiest when fed. features.extract_batch

~an order of magnitude on GPU ingest
Quantization

This very page runs the checkpoint in int8 (q8) — rankings barely move, downloads and memory shrink ~4×. The full-precision model stays offline.

4× smaller, near-identical rankings
Precompute the item side

Embeddings never change after ingest — so compute them offline, verify them with the agent, and serve rankings as one matrix multiply. item_tower.py

zero model calls at serving time
